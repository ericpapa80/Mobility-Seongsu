# 건축Hub 수집 데이터 활용 및 결합 방안

`수집체계_요약.md`와 OpenAPI 활용가이드를 기준으로, 수집된 **건축물대장·건축인허가** 데이터의 활용 방식과 **일대다(1:N)·다대일(N:1)** 관계가 있을 때의 결합(조인) 방안을 정리한 문서입니다.

---

## 1. 수집 산출물 구조

### 1.1 현재 산출 형식 (성수동 전용 등)

- **오퍼레이션별 별도 파일**로 저장됨. 하나의 통합·조인 파일은 생성하지 않음.
- 파일명 예: `{service}_{operation}_11200_seongsu_{timestamp}.json` / `.csv`
- 건축물대장·건축인허가별로 **동(성수동1가·2가)이 병합**된 단일 파일.

| 서비스 | 오퍼레이션(영문) | 국문 | 파일 예시 |
|--------|------------------|------|-----------|
| bldrgst | getBrBasisOulnInfo | 건축물대장 기본개요 | `bldrgst_getBrBasisOulnInfo_11200_seongsu_*.json` |
| bldrgst | getBrRecapTitleInfo | 건축물대장 총괄표제부 | (수집 시 별도 파일) |
| bldrgst | getBrTitleInfo | 건축물대장 표제부 | (수집 시 별도 파일) |
| bldrgst | getBrFlrOulnInfo | 건축물대장 층별개요 | (수집 시 별도 파일) |
| archpms | getApBasisOulnInfo | 건축인허가 기본개요 | `archpms_getApBasisOulnInfo_11200_seongsu_*.json` |

### 1.2 조인 시 공통 키 후보

- **건축물대장**: `mgmBldrgstPk` (관리건축물대장PK) — 가이드 및 첨부1 PK 전환 규칙 참고.
- **건축인허가**: 해당 API 응답의 대표 키(예: 인허가 고유 PK) — 활용가이드 응답 항목 참고.
- 시군구·법정동·번·지 등 조합으로 매칭할 경우 가이드의 항목명을 그대로 사용.

## 2. 관계 유형 정리

### 2.1 1:1 관계

- **예**: 건축물대장 기본개요 ↔ 총괄표제부 (건물 1건당 각 1건).
- **조인**: 공통 PK(예: `mgmBldrgstPk`) 기준으로 단순 조인 → 건물 1건당 1행, 컬럼만 확장.

### 2.2 1:N (일대다) / N:1 (다대일)

| 주체 | 다(N) 쪽 | 비고 |
|------|-----------|------|
| 건물 1건 | 층별개요 여러 행 | getBrFlrOulnInfo |
| 건물 1건 | 부속지번 여러 행 | getBrAtchJibunInfo |
| 건물 1건 | 전유부/호별 정보 여러 행 | getBrExposInfo 등 |
| 건물 1건 | 전유공용면적 여러 행 | getBrExposPubuseAreaInfo |

- 조인 시 **한 쪽 테이블이 여러 행으로 늘어나거나**, **별도 테이블로 두고 FK로 연결**하는 선택이 필요함.

---

## 3. 결합(조인) 방안

### 3.1 방안 A: 테이블(파일) 분리 + 조인 키 유지 (정규화, 권장)

- **구조**
  - `기본개요.csv`: 건물 1건당 1행, PK 포함.
  - `층별개요.csv`: 층 1건당 1행, **동일 PK(또는 FK) 컬럼** 포함.
  - `부속지번.csv`: 부속지번 1건당 1행, 동일 PK/FK 포함.
- **결합 시점**: DB 적재 후 SQL 조인, 또는 Pandas/Spark 등으로 **분석·사용 시점에** 조인.
- **장점**: 중복 없음, 1:N 관계가 그대로 유지됨, 용량·정합성 유리.
- **단점**: “한 파일에 모든 정보”가 아니어서, 사용 시 조인 쿼리·스크립트 필요.

**적용 예**

- 수집 단계: 현재처럼 오퍼레이션별 JSON/CSV 유지.
- 활용 단계: 공통 키(`mgmBldrgstPk` 등)를 FK로 두고, 필요한 조합만 조인해서 뷰/파일 생성.

---

### 3.2 방안 B: 넓은 테이블(디노멀라이즈)

- **구조**
  - 1:N 중 “다(N)” 쪽을 **한 행씩** 두고, 각 행에 **“1” 쪽 정보(건물명·주소 등)를 복제**해 붙임.
  - 예: 건물 A가 층 5개 → 5행, 5행 모두 동일한 건물 정보 컬럼 보유.
- **장점**: 하나의 flat CSV/테이블로 시각화·집계가 단순함.
- **단점**
  - 행 수·파일 크기 증가.
  - 층별·부속지번·호별을 **동시에** 하나의 테이블에 펼치면 행 수가 조합만큼 폭발하므로, **하나의 1:N만** 붙이는 것이 일반적 (예: 기본개요 + 층별개요만 wide 테이블).

**적용 예**

- 기본개요 + 층별개요만 선택해, `mgmBldrgstPk` 기준으로 left join 후 층별개요 행 수만큼 건물 정보 반복 저장.

---

### 3.3 방안 C: 계층(중첩) 구조

- **구조**
  - 건물 1건당 **하나의 문서(객체)** 로 두고, 1:N은 **배열**로 포함.
  - 예: `{ "mgmBldrgstPk": "...", "bldNm": "...", "floors": [ {...}, {...} ], "atchJibun": [ {...} ] }`
- **장점**: 1:N 관계가 스키마에 드러남, 문서 DB·API 응답에 적합.
- **단점**: CSV처럼 평탄한 테이블이 아니어서, R/Excel/일부 BI에서는 플래트닝(unnest) 등 추가 가공 필요.

**적용 예**

- 수집된 오퍼레이션별 JSON을 PK 기준으로 묶어, 건물 단위 JSON/GeoJSON 등으로 재구성하는 후처리 스크립트에서 사용.

---

## 4. 방안 선택 가이드

| 목적 | 추천 방안 |
|------|-----------|
| 일반적 저장·재사용·DB 적재 | **A (테이블 분리 + FK)** |
| 특정 분석용 단일 테이블 필요 | **B (해당 1:N만 wide 테이블)** |
| 문서/API 중심 활용 | **C (계층 구조)** |

- **일대다/다대일이 있는 결합**에서는, 기본은 **A**로 두고, 필요 시 B·C를 **뷰·전처리 결과**로 추가하는 구성을 권장함.

---

## 5. 활용 시 유의사항

- **PK 통일**: 건축물대장 쪽은 가이드·첨부1의 PK 전환 규칙에 따라 `mgmBldrgstPk` 등 표준 키를 일관되게 사용할 것.
- **인허가–대장 매칭**: 건축인허가와 건축물대장을 붙일 경우, 주소·지번·인허가번호 등 매칭 키는 활용가이드 응답 항목을 참고해 정의할 것.
- **수집 범위**: 현재 성수동 전용 기본 설정은 **기본개요만** 수집. 총괄표제부·표제부·층별개요·부속지번 등 전체 오퍼레이션 수집은 `scripts/archhub/collect_seongsu.py`의 `SEONGSU_DEFAULT_OPS` 확장 후 재수집 필요.

## 6. 공간데이터(VWorld)와의 결합 및 시각화

VWorld에서 수집한 **건물(도로명주소건물)·토지(LX맵)** 공간데이터와 건축물대장·건축인허가 데이터를 결합해 지도 위에 시각화하는 방안을 정리합니다.

### 6.1 데이터 소스 정리

| 구분 | 소스 | 좌표/형식 | 주요 결합 키 후보 |
|------|------|-----------|-------------------|
| 공간 | VWorld 도로명주소건물 (lt_c_spbd) | WFS → GeoJSON (EPSG:4326), `geometry` | `pnu`, 주소/도로명 |
| 공간 | VWorld 토지 (lt_c_landinfobasemap) | WFS → GeoJSON (EPSG:4326), `geometry` | `pnu`, `bldrgst_pk` |
| 속성 | 건축물대장 (bldrgst) | 시군구·법정동·번·지·대지위치, **무좌표** | `mgmBldrgstPk`, `platPlc`, 지번 |
| 속성 | 건축인허가 (archpms) | 주소·대지위치 등, **무좌표** | 인허가 PK, 주소·지번 |

- VWorld 건물/토지 수집 산출물: `collectors/data/raw/vworld/vworld_seongsu_*/` 하위 JSON·CSV·GeoJSON (레이어별).
- 건축Hub 산출물: `collectors/data/raw/archhub/archhub_seongsu_*/` 하위 대장·인허가 병합 JSON/CSV.

### 6.2 결합 키 및 전략

- **토지 ↔ 건축물대장**
  - VWorld 토지 레이어 속성 `bldrgst_pk`와 건축물대장 `mgmBldrgstPk`를 **동일 PK로 간주하고 조인** (가이드·전환규칙 확인 필요).  
  - 조인 시: 토지 폴리곤(GeoJSON)에 대장 속성(용도·면적·건물명 등)을 붙여 한 레이어로 시각화 가능.

- **PNU(필지고유번호)**
  - 토지: `pnu` 필드 보유 (일반적으로 19자리 등 표준 포맷).
  - 대장/인허가: `platPlc`, 시군구코드·법정동·번·지 조합으로 PNU를 **생성·정규화**하면 토지와 1:1 또는 N:1 조인 가능. (PNU 생성 규칙은 행정표준코드·가이드 참고.)

- **건물(SPBD) ↔ 대장/인허가**
  - 건물 레이어에는 `pnu`, `xpos`, `ypos`, 도로명·건물명 등이 있음.  
  - **방법 1**: 대장/인허가의 `platPlc`·`newPlatPlc`를 주소 정규화 후 건물 속성(도로명·지번)과 **문자열 매칭** 또는 퍼지 매칭.  
  - **방법 2**: 대장/인허가 주소를 **지오코딩**해 점(Point) geometry를 만든 뒤, VWorld 건물 폴리곤과 **공간 조인**(포함·가장 가까운 폴리곤)으로 연결.  
  - **방법 3**: 토지(bldrgst_pk) → 대장 조인 후, 건물과 토지를 **공간 조인**(건물이 속한 토지)해서 간접 연결.

### 6.3 결합 절차 (권장 순서)

1. **좌표계 통일**  
   VWorld 수집 시 GeoJSON은 EPSG:4326(WGS84)으로 저장. 다른 좌표계 데이터가 있으면 4326으로 변환.

2. **키 정규화**  
   - PNU: 자릿수·구분자 통일 (예: 19자리, 앞자리 0 패딩).  
   - `mgmBldrgstPk` vs `bldrgst_pk`: 문자열 공백·대소문자 등 통일 후 조인.

3. **조인 실행**  
   - 토지–대장: `bldrgst_pk` = `mgmBldrgstPk` (또는 PNU 기준) 로 테이블 조인.  
   - 건물–대장/인허가: 위 6.2의 문자열 매칭 또는 지오코딩+공간조인 중 하나 선택.

4. **시각화용 레이어 생성**  
   - 조인 결과를 **GeoJSON** 또는 **GeoPackage**로 저장 (geometry는 토지/건물 쪽 유지, 속성에 대장·인허가 컬럼 추가).  
   - 필요 시 건축물대장/인허가만 지오코딩해 **점 레이어**로 따로 그려 이중 시각화 가능.

### 6.4 시각화 방법

- **QGIS**  
  - 조인된 GeoJSON/GeoPackage를 레이어로 추가.  
  - 속성 테이블에서 대장·인허가 항목으로 **스타일(색·라벨)** 설정.  
  - 토지·건물·대장 점 레이어를 함께 올려 비교 가능.

- **웹 맵 (Leaflet / Mapbox GL 등)**  
  - 조인된 GeoJSON을 스크립트 또는 API로 불러와 레이어 추가.  
  - 팝업에 대장·인허가 필드 표시 (용도, 면적, 인허가일자 등).

- **주의**  
  - 일대다(한 필지에 건물 여러 동 등)가 있으면 조인 결과 행 수가 늘어남. 시각화 시 **중복 geometry** 제거 또는 대표 1건만 표시하는 옵션을 고려.

### 6.5 요약

| 목표 | 결합 대상 | 주요 키 | 산출물 예 |
|------|-----------|---------|-----------|
| 토지에 대장 정보 붙여 지도 표시 | VWorld 토지 + 건축물대장 | `bldrgst_pk` ↔ `mgmBldrgstPk` 또는 PNU | 토지 GeoJSON + 대장 컬럼 |
| 건물에 대장/인허가 붙이기 | VWorld 건물 + 대장/인허가 | 주소 매칭 또는 지오코딩+공간조인 | 건물 GeoJSON + 대장/인허가 컬럼 |
| 인허가만 지도에 점으로 표시 | 건축인허가 | - | 지오코딩된 Point GeoJSON |

- 수집 단계에서는 **공간·대장·인허가를 각각 별도 파일**로 유지하고, **후처리 스크립트**에서 위 키/전략으로 조인한 뒤 시각화용 GeoJSON을 만드는 구성이 유지·재현에 유리합니다.
